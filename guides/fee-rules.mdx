OwnPay uses a brand-scoped fee rules engine to calculate transactional commissions dynamically.

## How fee rules work

Fee rules are stored in `op_fee_rules` and resolved by `FeeRuleRepository::resolveActiveRule()`.

### Resolution priority

Among rules matching the exact currency, the most specific active rule wins:

1. **Specific Brand + Specific Gateway** (e.g., brand 5 + `bkash-api`)
2. **Specific Brand + Any Gateway** (`gateway_slug IS NULL`)
3. **Global + Specific Gateway** (`merchant_id IS NULL`)
4. **Global + Any Gateway**

Ties break by newest rule (`id DESC`).

### Currency matching

Currency is always an exact match. There is no "any currency" rule - every candidate must match the transaction currency exactly.

## Creating fee rules

1. Go to **System > Fee Rules**
2. Click **Add Rule**
3. Configure:
   - **Brand** - Apply to specific brand or all brands
   - **Gateway** - Apply to specific gateway or all gateways
   - **Currency** - Target currency code
   - **Fee Type** - Percentage, fixed, or tiered
   - **Fee Amount** - The commission rate
4. Click **Save**

## Fee calculation

At transaction time:

1. Resolve the most specific active rule for the brand, gateway, and currency
2. Apply the fee formula (percentage of amount, fixed fee, etc.)
3. Record the fee in the transaction
4. Post the ledger entry

## Schema

```sql
CREATE TABLE op_fee_rules (
    id INT AUTO_INCREMENT PRIMARY KEY,
    merchant_id INT NULL,
    gateway_slug VARCHAR(50) NULL,
    currency CHAR(3) NOT NULL,
    fee_type ENUM('percentage', 'fixed') NOT NULL,
    fee_amount DECIMAL(18,8) NOT NULL,
    is_active TINYINT(1) DEFAULT 1,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (merchant_id) REFERENCES op_merchants(id) ON DELETE CASCADE
);
```

## Best practices

- Start with global rules, then add brand-specific overrides as needed
- Test fee calculations with small transactions first
- Monitor fee revenue in financial reports
- Document your fee structure for audit purposes
