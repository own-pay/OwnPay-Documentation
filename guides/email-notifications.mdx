OwnPay sends transactional emails for payment events, password resets, and staff invitations. Emails are driven by the event system with sender identity resolved through the brand cascade.

## Email events

| Event | Trigger |
|-------|---------|
| `payment.transaction.completed` | Payment successful |
| `refund.created` | Refund processed |
| Password reset | User requests password reset |
| Staff invitation | New staff member invited |

## Configuration

### Per-brand email settings

1. Go to **System > Settings > Notifications**
2. Configure per brand:
   - **Admin notification email** - Where to send alerts
   - **Email on payment** - Toggle payment notifications
   - **Email on refund** - Toggle refund notifications

### Sender identity

Emails are sent from the brand's configured sender:

- **From Name** - Brand name or custom sender name
- **From Email** - Brand notification email

The cascade resolves: Brand override > All-Brands default > System default.

### SMTP configuration

1. Go to **System > Settings > Email**
2. Configure:
   - **SMTP Host** - e.g., `smtp.gmail.com`
   - **SMTP Port** - 587 (STARTTLS) or 465 (SSL)
   - **Username** - SMTP username
   - **Password** - SMTP password
   - **From Address** - Default sender email

## Email templates

Emails render from Twig templates in `templates/email/`:

- `payment_completed.twig` - Payment confirmation
- `password_reset.twig` - Password reset link
- `staff_invitation.twig` - Staff invitation

Customize templates per brand through the admin panel.

## Inheritance

Brand settings follow a cascade:

1. **Brand-specific override** - Set for this brand only
2. **All-Brands default** - Global default for all brands
3. **System default** - Hardcoded fallback

Leave fields blank/inherit to use the parent level.

## Best practices

- Test email delivery with a real SMTP provider
- Use a dedicated sending domain with proper SPF/DKIM
- Monitor email delivery logs for failures
- Keep notification templates professional and branded
